<p><strong>Code Walkthrough</strong></p>
<p>Each community deploys their own instance of the contract.</p>
<p>The constructor allows all parameters to be customised for this particular community. The settings can also be changed later.</p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code>constructor(
string memory _name,
string memory _symbol,
uint8 _decimals,
uint _lifetime,
uint _generationAmount,
uint _generationPeriod,
uint _communityContribution,
uint _transactionFee,
uint _initialBalance,
address _communityContributionAccount,
address _controller
)
</code></pre><p>By default, the account deploying the contract has permission to change settings. This can be changed to a community multisig contract, or handled by Aragon. (see Aragon branch). We refer to the account or contract with this permission as the controller.</p>
<p>Any account can receive tokens. To receive the UBI generation, an account first has to be registered by the controller.</p>
<p><strong>Creating an account:</strong></p>
<hr>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code> /** @notice Create a new account with the specified role
@dev New accounts can always be created.
This function can&apos;t be disabled. */
function verifyAccount(
address _account
)
external
onlyController
{
accountApproved[_account] = true;
if (initialBalance &gt; 0 &amp;&amp; lastTransactionBlock[_account] == 0) {
_mint(_account, initialBalance);
emit Mint(_account, initialBalance);
zeroBlock[_account] = block.number.add(lifetime);
}
lastGenerationBlock[_account] = block.number;
lastTransactionBlock[_account] = block.number;
emit VerifyAccount(_account);
}
</code></pre><p>The account is marked as approved.</p>
<p>As long as there are no previous transactions, the initial balance is minted. This check ensures accounts can&apos;t receive multiple initial balance payments.</p>
<p>The zero block is set to the full lifetime.</p>
<p>The last generation block is set to the current block.</p>
<p>The last transaction block is set to the current block.</p>
<hr>
<p><strong>Transfering tokens</strong></p>
<p>Most of the logic is in the transfer function.</p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code> /** @notice Transfer the currency from one account to another,
updating each account to reflect the time passed, and the
effects of the transfer
@return Success */
function transfer(address _to, uint _value) public returns (bool) {
// Process generation and decay for sender
triggerOnchainBalanceUpdate(msg.sender);
// Process generation and decay for recipient
triggerOnchainBalanceUpdate(_to);
require(
_balances[msg.sender] &gt;= _value,
&quot;Not enough balance to make transfer&quot;
);
// Process fees and contribution
uint feesAndContribution = processFeesAndContribution(
_value,
transactionFee,
communityContribution
);
uint valueAfterFees = _value.sub(feesAndContribution);
//Extend zero block based on transfer
zeroBlock[_to] = calcZeroBlock(
valueAfterFees,
_balances[_to],
block.number,
lifetime,
zeroBlock[_to]
);
/* If they haven&apos;t already been updated (during decay or generation),
then update the lastTransactionBlock for both sender and recipient */
if (lastTransactionBlock[_to] != block.number) {
lastTransactionBlock[_to] = block.number;
}
if (lastTransactionBlock[msg.sender] != block.number) {
lastTransactionBlock[msg.sender] = block.number;
}
super.transfer(_to, valueAfterFees);
return true;
}
</code></pre><p>First, we update the balance of both sender and recipient, to take into account the time passed since the last transaction. This is handled by <code>triggerOnchainBalanceUpdate</code></p>
<p><em>Trigger on chain balance update</em></p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code> /// @notice Manually trigger an onchain update of the live balance
function triggerOnchainBalanceUpdate(address _account) public {
// 1. Decay the balance
uint decay = calcDecay(
lastTransactionBlock[_account],
_balances[_account],
block.number,
zeroBlock[_account]
);
uint decayedBalance;
if (decay &gt; 0) {
decayedBalance = burnAt(_account, decay);
emit Decay(_account, decay);
} else {
decayedBalance = _balances[_account];
}
// 2. Generate tokens
uint generationAccrued = 0;
if (accountApproved[_account]) {
// Calculate the accrued generation, taking into account decay
generationAccrued = calcGeneration(
block.number,
lastGenerationBlock[_account],
lifetime,
generationAmount,
generationPeriod
);
if (generationAccrued &gt; 0) {
// Issue the generated tokens
_mint(_account, generationAccrued);
emit IncomeReceived(_account, generationAccrued);
// Record the last generation block
lastGenerationBlock[_account] += calcNumCompletedPeriods(
block.number,
lastGenerationBlock[_account],
generationPeriod
).mul(generationPeriod);
// Extend the zero block
zeroBlock[_account] = calcZeroBlock(
generationAccrued,
decayedBalance,
block.number,
lifetime,
zeroBlock[_account]
);
}
}
// Record the last transaction block
if (decay &gt; 0 || generationAccrued &gt; 0) {
lastTransactionBlock[_account] = block.number;
}
}
</code></pre><p>First we calculate whether any tokens have decayed, and then we burn those tokens.</p>
<p>If the account has been approved, we then calculate the generation that is due, and mint the tokens to the account&apos;s balance. Tokens will only be issued for each full generation period that has elapsed since the last transfer.</p>
<p>The last generation block is then recorded. Last generation block will be increased in multiples of the generation period, starting with the block that the account was approved. A new account will receive income for the first time after a full generation period has passed following its creation.</p>
<p>We then calculate the effect that this incoming transfer has on the <code>zeroBlock</code>. The generation counts as an incoming transfer and thus extends the lifetime of this account&apos;s balance.</p>
<p>As long as there has been some decay or generation, we update the <code>lastTransactionBlock</code> to the current block.</p>
<hr>
<p><em>Transfer (continued)</em></p>
<p>Once both onchain balances reflect the current block, (which represents the current date), we go ahead and process the transfer.</p>
<p>First we check the sender has enough balance to make the transfer.</p>

</code></pre><p>Next, we process the fees and contribution</p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code>// Process fees and contribution

<p><em>Process fees and contribution</em></p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code> /** @notice Calculate the fees and contribution, send contribution to the communtiy account,
and burn the fees
@dev Percentage to x dp as defined by contributionFeeDecimals e.g.
when contributionFeeDecimals is 2, 1200 is 12.00%
@return The total amount used for fees and contribution */
function processFeesAndContribution(
uint _value,
uint _transactionFee,
uint _communityContribution)
internal
returns (uint)
{
uint feesIncContribution = calcFeesIncContribution(
_value,
_transactionFee
);
uint contribution = calcContribution(
_value,
_transactionFee,
_communityContribution
);
uint feesToBurn = calcFeesToBurn(
_value,
_transactionFee,
_communityContribution
);

&quot;feesIncContribution should equal contribution + feesToBurn&quot;

}
</code></pre><p>First the fee is calculated. The contribution is taken as a percentage of this fee.</p>
<p>contributionFeeDecimals defines the number of decimal places in the percentages. So, if contributionFeeDecimals is 2,<br>
a fee of 3300 and contribution of 1000 indicates a fee of 33%, with 10% of this fee taken as contribution.</p>
<p>With these values, a transfer of 1000 would incur a fee of 330, with 33 sent to the community account as contribution, and 297 burned.</p>

<p>Next, we extend the zero block for the recipient.</p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code>//Extend zero block based on transfer

<p><em>Calculate Zero Block</em></p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code> /** @notice Calculate the block at which the balance for this account will
be zero
@return Block at which balance is 0 */
function calcZeroBlock(
uint _value,
uint _balance,
uint _blockNumber,
uint _lifetime,
uint _originalZeroBlock
)
public
pure
returns (uint)
{
if (_balance == 0 || _originalZeroBlock == 0) {
// No other transaction to consider, so use the full lifetime
return _blockNumber.add(_lifetime);
}
/* transactionWeight is the ratio of the transfer value to the total
balance after the transfer */
uint transactionWeight = _value.mul(multiplier).div(
_balance.add(_value)
);
/* multiply the full lifetime by this ratio, and add
the result to the original zero block */
uint newZeroBlock = _originalZeroBlock.add(
_lifetime.mul(transactionWeight).div(multiplier)
);
if (newZeroBlock &gt; _blockNumber.add(_lifetime)) {
newZeroBlock = _blockNumber.add(_lifetime);
}
return newZeroBlock;
}
</code></pre><p>All calculations are pure functions to make testing easier.</p>
<p>If this is the first transaction, or the balance is 0, we give the balance the full lifetime.</p>
<p>Otherwise, we add a portion of the full lifetime, to the same ratio as the weight of the transfer. [the ratio of the transfer value to the total balance after the transfer].</p>
<p>This means that a small transfer (compared to the balance) will extend the total lifetime by a small amount. Whereas, a large transfer will extend the total lifetime by a large amount. blocksToZero will never exceed the value of the current block plus the lifetime setting.</p>
<hr>
<p><em>Transfer (continued)</em></p>
<p>Next, we update the lastTransactionBlock for recipient and sender. This probably already occured during triggerOnchainBalance update, as there was probably some decay. If it&apos;s already the current block, we don&apos;t waste gas by setting it again.</p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code>/* If they haven&apos;t already been updated (during decay or generation),
then update the lastTransactionBlock for both sender and recipient */
if (lastTransactionBlock[_to] != block.number) {
lastTransactionBlock[_to] = block.number;
}
if (lastTransactionBlock[msg.sender] != block.number) {
lastTransactionBlock[msg.sender] = block.number;
}
</code></pre><p>Finally, we trigger the transfer of tokens, minus the fees and contribution. ZeppelinOS ERC20 implementation handles the transfer.</p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code>super.transfer(_to, valueAfterFees);
</code></pre><p><strong>Calculating Decay</strong></p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code> /** @notice Calculate the number of tokens decayed since the last transaction
@return Number of tokens decayed since last transaction */
function calcDecay(
uint _lastTransactionBlock,
uint _balance,
uint _thisBlock,
uint _zeroBlock
)
public
pure
returns (uint)
{
require(_thisBlock &gt;= _lastTransactionBlock,
&quot;Current block must be &gt;= last transaction block&quot;
);
// If zero block has not been set, decay = 0
if (_zeroBlock == 0) {
return 0;
}
// If zero block passed, decay all
if (_thisBlock &gt;= _zeroBlock) {
return _balance;
}
// If no blocks passed since last transfer, nothing to decay
uint blocksSinceLast = _thisBlock.sub(_lastTransactionBlock);
if (blocksSinceLast == 0) {
return 0;
}
/* Otherwise linear burn based on &apos;distance&apos; moved to zeroblock since
last transaction */
uint fullDistance = _zeroBlock.sub(_lastTransactionBlock);
uint relativeMovementToZero = blocksSinceLast.mul(
multiplier
).div(fullDistance);
return _balance.mul(relativeMovementToZero).div(multiplier);
}
</code></pre><p>Calculating the decay assumes that the zeroBlock is set correctly, and that the balance was accurate as of the last transaction block.</p>
<p>If no zero block has been set, there will be no decay.</p>
<p>If the current block is greater than the zero block, all of the balance has decayed.</p>
<p>If the lastTransactionBlock is the current block, the balance is up to date and nothing further to decay.</p>
<p>Assuming none of these conditions have been met, we perform a linear calculation taking into account the relative movement to the zero block, since the last transaction block. So, if at the last transaction block we were 100 blocks from the zeroBlock, and at this block we are only 50 blocks from the zeroBlock, 50% of the previous balance needs to decay.</p>
<p><strong>Calculating generation</strong></p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code>/** @notice Calculate the generation accrued since the last generation
period
@dev This function contains a for loop, so in theory may fail for
accounts that have been inactive for an extremely long time. These
accounts will have zero balance anyway. */
function calcGeneration(
uint _blockNumber,
uint _lastGenerationBlock,
uint _lifetime,
uint _generationAmount,
uint _generationPeriod
)
public
pure
returns (uint)
{
uint numCompletePeriods = calcNumCompletedPeriods(
_blockNumber,
_lastGenerationBlock,
_generationPeriod
);
uint decayPerBlock = multiplier.div(_lifetime);
uint decayPerGenerationPeriod = decayPerBlock.mul(_generationPeriod);
uint remainingPerGenerationPeriod = multiplier.sub(
decayPerGenerationPeriod
);
uint generation = 0;
for(uint i = 0; i &lt; numCompletePeriods; i++) {
generation = generation.mul(
remainingPerGenerationPeriod
).div(multiplier).add(_generationAmount);
}
return generation;
}
</code></pre><p>The generation amount is issued in full after each generation period.</p>
<p>The periods are counted from when the account is first created, so accounts will receive generation on different blocks. Accounts will never receive partial payments of the generation amount.</p>
<p>Generation is only calculated when an account sends or receives a transfer. Depending on the activity of each account, many generation periods may have passed in between transfers. When there are multiple periods, the function iterates through each generation period, applies the decay that would have taken place during the last period, and then adds the next generation amount. This design ensures that the generation income is the same regardless of account activity, while saving gas by not having to update the balance of unused accounts. As soon as a dormant account is used, all generation periods it missed will be processed in one block.</p>
<p><strong>Live Balance Of</strong></p>
<pre data-role="codeBlock" data-info="solidity" class="language-solidity"><code>/** @notice Return the real balance of the account, as of this block
@return Latest balance */
function liveBalanceOf(address _account) public view returns (uint) {
uint decay = calcDecay(
lastTransactionBlock[_account],
_balances[_account],
block.number,
zeroBlock[_account]
);
uint decayedBalance = _balances[_account].sub(decay);
if (lastGenerationBlock[_account] == 0) {
return(decayedBalance);
}
uint generationAccrued = calcGeneration(
block.number,
lastGenerationBlock[_account],
lifetime,
generationAmount,
generationPeriod
);
return decayedBalance.add(generationAccrued);
}
</code></pre><p><code>liveBalanceOf</code> should be used by GUIs in place of the <code>balanceOf</code> function, which will return an outdated balance. <code>liveBalanceOf</code> will calculate the up to date balance including generation and decay.</p>
<p><code>balanceOf</code> will show the correct balance as of the <code>lastTransactionBlock</code></p>
<p>Alternatively, <code>triggerOnchainBalance</code> can be called to force an update. This will incur gas, but after this balanceOf will return the up to date value.</p>
